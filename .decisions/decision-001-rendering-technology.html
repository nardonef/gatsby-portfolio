Rendering Technology

Technical

This is the foundation everything else builds on. The rendering approach determines what visual effects are possible, how interactive we can get, and how it performs on phones. Your current setup is pure CSS animations on SVG circles with a gooey filter — simple but limited. We need something that can handle organic movement, real-time mouse tracking, and color shifting without dropping frames.

Use the HTML Canvas 2D API with a classic metaball algorithm (computing an energy field per-pixel from multiple point sources). Blobs naturally merge and separate based on proximity — no SVG filter hack needed. Mouse position becomes another energy source that pulls or repels the blobs. Zero dependencies. You write the rendering loop yourself with requestAnimationFrame. This is the most "from scratch" option.

Works well when

  • You want full pixel-level control
  • Zero external dependencies
  • True metaball physics (natural merge/split)

Watch out for

  • Per-pixel calculation is CPU-heavy on large screens
  • Harder to get smooth gradients and glow effects

Metaball Field

Write a GLSL fragment shader that runs on the GPU. Every pixel is computed in parallel, so you get buttery 60fps even on full-screen canvases. The metaball energy field, noise-based organic distortion, color gradients, and glow effects all happen in a single shader pass. Mouse position and time are passed in as "uniforms" (variables the shader reads each frame). This is the same tech behind sites like Stripe's gradient and the liquid effects on award-winning portfolios. Thin wrapper — no Three.js needed, just raw WebGL with a fullscreen quad.

Works well when

  • GPU-accelerated: 60fps at any resolution
  • Noise, glow, color blending are trivial in GLSL
  • Highest visual ceiling of all options
  • ~50 lines of JS setup + the shader code

Watch out for

  • GLSL is a different language (but the JS wrapper is tiny)
  • WebGL context can fail on very old devices (graceful fallback needed)

Use Three.js as a convenience layer over WebGL. You still write the same GLSL shader for the blob effect, but Three.js handles the WebGL boilerplate (context creation, resize handling, render loop). It's the standard choice for WebGL in production. The downside: you're pulling in a ~150KB library mostly to avoid writing ~50 lines of WebGL setup code. Makes more sense if you plan to add 3D elements later (like depth, lighting, or 3D blob geometry).

Works well when

  • Familiar API if you know Three.js
  • Easy to extend with 3D later
  • Handles edge cases (context loss, resize)

Watch out for

  • ~150KB dependency for a 2D effect
  • Overkill — you'd use 5% of the library

Keep the SVG + gooey filter approach but replace CSS keyframes with GSAP (a professional animation library). GSAP can do noise-based path morphing, spring physics, and mouse-following with its built-in easing. This is the least disruptive change — you're enhancing what you have rather than replacing it. But the SVG filter approach has a hard ceiling: it can't do per-pixel glow, smooth color gradients within the blob, or the fluid look that shaders can achieve.

Works well when

  • Smallest change from current code
  • GSAP is battle-tested and well-documented
  • SVG stays accessible and inspectable

Watch out for

  • SVG filter performance degrades with complexity
  • Hard ceiling on visual effects (no glow, no gradients inside blobs)
  • Won't achieve the "wow" factor of GPU-rendered effects
